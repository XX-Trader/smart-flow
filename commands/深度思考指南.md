# Deepdive 使用指南

## 🚀 快速开始

```bash
# 三种方式启动深度思考

/深度思考 如何设计推荐系统？

/深度分析 我们应该用什么数据库？

/深度迭代 产品定价策略如何制定？
```

---

## 📋 核心差异对比

### 传统模式（表演式）

```
用户: 如何设计通知系统？

AI: 好的！推荐方案如下：
    1. 实时通知：WebSocket
    2. 队列：Redis + Bull
    3. 优先级：高/中/低三级
    ...

问题：❌ 没问业务场景
     ❌ 没问用户量
     ❌ 没问实际需求
```

### Deepdive 模式（真实思考）

```
用户: 如何设计通知系统？

AI: 等等，先别急着给方案...

困惑：
- "通知"的业务目标是什么？
- 用户量多大？日活多少？
- 现有数据吗？

探索A：从用户场景理解
- MVP阶段（<1000用户）：可能邮件就够了
- 增长期（1万-10万）：需要站内通知
- 成熟期（10万+）：需要实时推送

探索B：从产品类型理解
- 社交产品：实时性强
- 工具产品：弱实时性
- 内容产品：按需推送

反例验证：
- 用户100个却上Kafka → 杀鸡用牛刀
- 全部实时推送 → 用户被打扰

诚实结论：
完成度：60%

需要先确认：
1. 当前用户量？
2. 产品类型？
3. 核心使用场景？

分阶段建议：
- V1：简单站内通知
- V2：邮件通知 + 分类
- V3：实时推送

待解决：防骚扰策略、退订机制...
```

## 🎯 适用场景

| 场景 | 是否适合 | 原因 |
|-----|---------|------|
| 架构设计 | ✅ | 没有标准答案 |
| 技术选型 | ✅ | 需要多角度权衡 |
| 产品方向 | ✅ | 需要理解本质 |
| 性能优化 | ✅ | 需要深入分析 |
| Bug修复 | ❌ | 有明确方案 |
| 代码实现 | ❌ | 技术执行 |
| 信息查询 | ❌ | 无需设计 |

## 💡 使用技巧

### 1. 识别何时需要深度思考

```
✅ 需要深度思考的信号：
- "应该如何设计..."
- "怎么选择..."
- "有什么建议..."
- "最佳实践是什么..."

❌ 不需要的信号：
- "如何实现XX功能"（实现问题）
- "XX的API是什么"（查询问题）
- "帮我修复这个bug"（明确问题）
```

### 2. 配合使用

```
# 深度思考 + 自动循环 = 强力组合
/深度思考 设计认证方案
→ 分析完后
/循环 实现认证功能 次数 20 完成 测试通过
```

### 3. 预期效果

```
时间投入：普通回答的 3-5 倍
思考深度：显著提升
结论质量：更可靠、更全面
用户参与：需要回答澄清问题
```
