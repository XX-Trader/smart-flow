## 📖 什么是斜杠命令？

斜杠命令（Slash Commands）是快速调用特定功能的快捷方式。只需输入 `/命令名` 即可启动，无需完整描述需求。

**示例**:
```
/commit                           # 创建规范的 Git 提交
/深度思考 如何设计推荐系统？      # 启动深度思考模式
/循环 让测试通过                  # 自动循环执行任务
```

## 🚀 可用命令列表

### 1. `/commit` - Git 提交命令

**功能**: 创建符合 Conventional Commits 规范的提交信息

**使用方式**:
```bash
/commit                                    # 基本用法
/commit --no-verify                        # 跳过预提交检查
/commit --style=full                       # 详细提交信息
/commit --style=full --type=feat           # 指定提交类型
```

**参数**:
- `--no-verify`: 跳过预提交检查（lint, build, docs）
- `--style=simple|full`: simple=简洁单行, full=详细多行
- `--type=<type>`: 指定类型 (feat|fix|docs|style|refactor|perf|test|chore|ci|build|revert)

**支持的提交类型**:
| 类型 | Emoji | 说明 |
|------|-------|------|
| feat | ✨ | 新功能 |
| fix | 🐛 | Bug 修复 |
| docs | 📝 | 文档更新 |
| style | 🎨 | 代码格式 |
| refactor | ♻️ | 重构 |
| perf | ⚡ | 性能优化 |
| test | ✅ | 测试 |
| chore | 🔧 | 维护 |
| ci | 👷 | CI/CD |
| build | 📦 | 构建系统 |
| revert | ⏪ | 回滚 |

**完整文档**: [commit.md](./commit.md)

### 2. `/深度思考` - 深度思考模式

**功能**: 让 AI 真正深度思考，而非表演回答

**使用方式**:
```bash
/深度思考 如何设计推荐系统？
/深度思考 技术选型：Go vs Rust？
/深度思考 产品定价策略如何制定？
```

**特点**:
- 🧠 **真实思考**: 允许困惑、探索、推翻重来
- 🔍 **多角度**: 用户场景/技术实现/业务目标
- ✅ **案例验证**: 用完整场景测试结论
- 📊 **诚实总结**: 承认未完成部分和边界

**适用场景**:
- ✅ 架构设计
- ✅ 技术选型
- ✅ 产品方向
- ✅ 性能优化
- ❌ Bug 修复（有明确方案）
- ❌ 代码实现（技术执行）
- ❌ 信息查询（无需设计）

**完整文档**: [深度思考指南.md](./深度思考指南.md)

### 3. `/深度分析` - 多角度分析

**功能**: 多角度探索问题本质

**使用方式**:
```bash
/深度分析 这个架构设计合理吗？
/深度分析 为什么性能会下降？
```

**探索框架**:
1. 问题本质 - 真正要解决什么？
2. 探索角度 - 用户场景/技术实现/业务目标
3. 反例验证 - 不这样会怎样？
4. 案例测试 - 完整场景验证
5. 诚实总结 - 关键洞察 + 待解决问题

**特点**: 可能需要 3-5 轮迭代，每轮可能推翻上一轮的理解

### 4. `/循环` - 自动循环执行

**功能**: AI 自动循环执行任务直到完成

**使用方式**:
```bash
# 交互式（推荐新手）
/循环 让测试通过

# 一行搞定
/自动循环 修复测试 次数 20 完成 测试全部通过

# 更口语化
/反复 实现登录功能
```

**需要回答 3 个问题**:
1. 需要反复吗？
2. 最多几次？
3. 何时算完成？

**示例**:
```
你: /循环 让测试通过

我: 1. 需要反复尝试吗？(是/否)
   2. 最多尝试几次？
   3. 怎么算完成？

你: 是, 20, 测试全部通过

我: 开始自动循环！
```

**完整文档**: [使用说明.md](./使用说明.md)

### 5. `/自动循环` - 一键启动循环

**功能**: 一行命令启动自动循环

**使用方式**:
```bash
/自动循环 <任务> 次数 <数字> 完成 <完成条件>
```

**示例**:
```bash
/自动循环 修复Bug 次数 10 完成 所有测试通过
/自动循环 优化性能 次数 15 完成 响应时间<100ms
```

**优势**: 无需交互，一行搞定

### 6. `/反复` - 反复执行

**功能**: 反复执行直到完成（更口语化）

**使用方式**:
```bash
/反复 修复登录问题
/反复 优化代码
```

**特点**: 与 `/循环` 功能相同，更口语化的命令名

## 💡 使用技巧

### 1. 组合使用

```
# 深度思考 + 自动循环
/深度思考 设计认证方案
→ 分析完后
/循环 实现认证功能 次数 20 完成 测试通过
```

### 2. 选择合适的命令

```
场景：需要创建 Git 提交
├─ 简单提交 → /commit
├─ 详细提交 → /commit --style=full
└─ 跳过检查 → /commit --no-verify

场景：需要解决复杂问题
├─ 架构设计 → /深度思考
├─ 多角度分析 → /深度分析
└─ 迭代实现 → /循环

场景：需要反复尝试
├─ 想讨论次数 → /循环
├─ 一行搞定 → /自动循环
└─ 更口语化 → /反复
```

### 3. 快速参考

```
Git 提交:        /commit
深度思考:        /深度思考 <问题>
多角度分析:      /深度分析 <问题>
交互式循环:      /循环 <任务>
一行循环:        /自动循环 <任务> 次数 <N> 完成 <条件>
口语化循环:      /反复 <任务>
```

## 🎯 典型工作流

### 工作流 1: 功能开发

```bash
# 1. 深度思考设计
/深度思考 如何设计用户认证系统？

# 2. 自动循环实现
/循环 实现认证功能 次数 20 完成 测试通过

# 3. 创建提交
/commit --style=full --type=feat
```

### 工作流 2: Bug 修复

```bash
# 1. 分析问题
/深度分析 为什么会出现这个Bug？

# 2. 循环修复
/反复 修复测试失败 次数 15 完成 所有测试通过

# 3. 提交
/commit --type=fix
```

### 工作流 3: 性能优化

```bash
# 1. 深度分析瓶颈
/深度分析 性能瓶颈在哪里？

# 2. 循环优化
/自动循环 优化查询 次数 10 完成 响应时间<100ms

# 3. 提交
/commit --type=perf --style=full
```
